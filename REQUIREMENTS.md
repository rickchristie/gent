I want to work on ToolSearchToolChain. This should be implementable using our ToolChain and Tool interfaces. ToolSearchToolChain can have many, many tools registered, but it always presents 1 single tool in the available tools section: "Tool Registry Search". This tool is built-in and isn't provided by the user.

Tool Registry Search tool has complex, modifiable description that also acts as prompt. Based on research, this tool prompt has to give LLM 3 things:
- Domain Awareness (what categories of tool exists)
- Scale Awareness (how many tools roughly)
- Search Guidance (how to formulate good queries when searching tools)

I'm thinking that we can create IndexableTool interface. It has additional methods that ToolSearchToolChain uses.
Tools that is registered to ToolSearchToolChain MUST implement IndexableTool interface.

```
type IndexableTool interface {
  // Name mirrors [Tool.Name].
  Name() string

  // Description mirrors [Tool.Description].
  Description() string

  // Domain returns the high level domain that this tool belongs to.
  Domain() string

  // Categories returns list of categories in this Domain that applies to this tool.
  Categories() []string

  // Keywords are short keywords of verb (action), noun, topic, capabilities, or anything
  // that can be added to help this tool accurately surface when needed.
  Keywords() []string

  // SyntheticQueries is implementation of Reverse-HyDE (Hypothetical Document Embeddings) where 
  // hypothetical queries are generated for each document in the corpus. Each string returned is
  // generated by asking, "what queries would a use ask that would need this tool?".
  //
  // This technique works well in RAG search, not recommended for non-RAG searches. Why not auto
  // generate this query? Because ToolSearchToolChain doesn't persist, if we auto generate it,
  // each time we start the app, we'll generate different queries, which makes it harder to
  // optimize.
  SyntheticQueries() []string
}
```

Another abstraction is to differentiate the search engine with ToolSearchToolChain - meaning we can defer building actual RAG system later on, and just plug it in without changing the entire ToolSearchToolChain interface, I'm thinking something like:

```
type SearchEngine interface {
   // Id used to differentiate this engine with others.
   // Convention is to use lowercase snakecase.
   Id() string
  
   // SearchGuidance is the prompt given to LLM to guide them in writing effective search queries.
   // The Re-Invoke paper (EMNLP 2024) showed that query formulation is the bottleneck, not retrieval quality.
   // Prompt should include 2-3 examples of good search queries, not to list keywords, but to teach LLM how to search effectively.
   SearchGuidance() string

   // IndexAll takes all tools and index them for searching.
   IndexAll(tools []IndexableTool) error

   // Search takes in a query and return list of tool names ([Tool.Name]) that matches the query.
   // When error is returned, this error message is surfaced to the LLM, so make sure to return helpful error messages
   // for dynamic steering. Search returns ALL matching results, ranked by relevance.
   // [ToolSearchToolChain] takes care of pagination.
   Search(ctx context.Context, query string) ([]string, error)
}
```

The query_type enum is not hardcoded, it's generated based on registered SearchEngine.
For the SearchEngine implementation, we create two out-of-the-box implementations:
- Bm25SearchEngine implementation that uses Bleve with NewMemOnly() with minimal custom mapping that only indexes fields that we search.
- RegexSearchEngine implementation uses Golang regex.
This means users just need to create their Tools with appropriate keywords, descriptions, and register them.

Why Bleve?
- None of other BM25 implementations in Go seem to be active in development, or even used in production.
- Much better multi-language support.
- Stable. Bleve has the support of couchbase, used in production, active in development.

ToolSearchToolChain requires number of results per page to be configured, default to 3.
SearchEngine might match a ton of tools ranked, so without pagination, it will be expensive for LLM to process all results, and it will reduce LLM accuracy as well.
Pagination allows LLM to focus on top results, and if needed, it can ask user to refine query or ask for next page.

BM25 indexes the following:
- High boost: Name, Description
- Medium boost: Keywords, Categories
- Lower boost: Domain, SyntheticQueries

Regex matches against:
- Name, Description, Keywords, Categories, Domain, SyntheticQueries
It ranks by counting number of matches, for ties - there's no specific order.

I think we only need to create ToolSearchToolChainJson - no need to create YAML version for now. 
Tool Registry Search tool is called like any other tool, when called, it will run SearchEngine and return full tool definition that matches the query.
The tool definition is displayed just like how it would be displayed in non-search "available tools" section, so name, description, policy, parameter schemas are displayed.

ToolSearchToolChain has `Initialize() error` method that reindex all registered tools and computes all Domain, categories, number of tools, etc. 
Initialize() MUST be called before it's used, it's responsibility of user to call Initialize() before using ToolSearchToolChain.
Initialize() can be called multiple times, and each call will reindex the tools and update the internal state (mutex might be needed here).

ToolSearchToolChain publish events and tracks stats just like other ToolChain implementation.

ToolSearchToolChain is just like other ToolChain implementation, with the difference of:
- Having a built-in "Tool Registry Search" tool that is always available.
- Hides all registered tools and only surface "Tool Registry Search" in the available tools section.
ToolSearchToolChain parses and executes all registered tool calls, it just hides them from available tools section and gives LLM tool to search for them.

Just like current ToolChain implementation, ToolSearchToolChain allows parallel tool call. 
Tool Registry Search itself is a tool call, so LLMs can run multiple searches in one iteration.
This allows LLM to call search tool and tool calls in parallel in the same iteration.

ToolSearchToolChain surfaces an error message if no results are found, this string can be configured by the user.
By default it just says "No tools found matching your query. Please refine your query and try again.".
